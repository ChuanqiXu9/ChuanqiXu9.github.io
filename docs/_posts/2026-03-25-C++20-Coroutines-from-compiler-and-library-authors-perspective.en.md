# Clang/LLVM's Implementation Choice for Coroutines

Compilers can be simply divided into:

- Frontend: Language-specific parts.
- Midend: Language-agnostic and architecture-agnostic parts. Mainly various reusable optimizations.
- Backend: Architecture-specific parts.

Most language features are basically implemented solely in the frontend. C++ committee compiler engineers are also mostly frontend engineers (from my impression, though I haven't actually verified this). When frontend engineers implement language features, they naturally choose to complete the implementation in the frontend. This is completely reasonable.

However, when Clang implemented Coroutines TS, Gor Nishanov astutely discovered that implementing C++20 Coroutines in the midend can greatly enhance the performance potential of C++20 Coroutines.

My favorite example: https://godbolt.org/z/ad9TjqGan:

![Clang vs GCC Coroutine Implementation Comparison](/images/Coro-Clang-VS-GCC.png)

This approximately 100-line C++20 coroutine code is optimized by the Clang compiler to only 3 lines of assembly, while with the same O3 optimization enabled, GCC generates over 600 lines of code.

Of course, this example might be too extreme. However, in production, we have also observed that code generated by Clang for C++20 coroutines is significantly better than code generated by GCC. A coroutines-heavy application compiled with Clang showed approximately 20% performance improvement in end-to-end comparison with GCC-compiled output. In traditional compilation, this is a very significant performance improvement.

There has always been a saying that GCC is always better than Clang in terms of compiled output performance (of course, I don't fully agree with this statement alone, and it could be discussed elsewhere). The C++20 coroutines example shows that for new language features, even in runtime performance, GCC doesn't always outperform Clang. I also think language-agnostic midend optimization could be an excellent direction, though that's a bigger topic.

The main reason for the performance difference between GCC and Clang in C++20 coroutines is that GCC chose to implement C++20 coroutines in the frontend, while Clang chose to implement C++20 coroutines collaboratively between frontend and midend.

There are two key points for C++20 coroutine compiler support:

- Coroutine frame generation
- Coroutine function splitting

C++20 coroutines are stackless coroutines, which are functions that can be paused. To support pauseable functions, we need to save the intermediate state when the function is paused. This intermediate state is the coroutine frame.

The key difference between frontend implementation and midend implementation is whether we generate the coroutine frame before or after optimization occurs.

Here's a constructed case:

```C++
#include <coroutine>
#include <iostream>
#include <cstdlib>

struct Task {
    struct promise_type {
        void* operator new(std::size_t size) {
            std::cout << "Coroutine allocated " << size << " bytes\n";
            return ::operator new(size);
        }

        Task get_return_object() { 
            return Task{std::coroutine_handle<promise_type>::from_promise(*this)}; 
        }
        std::suspend_always initial_suspend() noexcept { return {}; }
        std::suspend_always final_suspend() noexcept { return {}; }
        void return_void() {}
        void unhandled_exception() {}
    };

    std::coroutine_handle<promise_type> handle;

    ~Task() { 
        if (handle) handle.destroy(); 
    }

    bool done() const {
        return handle && handle.done();
    }
};

[[clang::noinline]]
Task my_coro() {
    co_await std::suspend_always{};
    co_return;
}

int main() {
    auto t = my_coro();
    while (!t.done())
        t.handle.resume();
    return 0;
}
```

In this case, we record the coroutine frame allocation size in `Task::promise_type::operator new` to observe compiler optimization. Note that I need to use `[[clang::noinline]]` on `my_coro()` to disable Clang optimization; otherwise, we wouldn't see anything. GCC doesn't recognize `[[clang::noinline]]` anyway.

The result of running the above code directly is:

```

```

This is likely because GCC saves `std::suspend_always{}` to the coroutine frame, causing a larger frame.

In short, while not strictly correct but perhaps not particularly wrong, a conclusion could be that coroutine functions under GCC basically don't receive much optimization. Based on this, finding examples that Clang can optimize is very easy to construct shocking optimization result comparisons between Clang and GCC. Saying that Clang's coroutine performance is N times that of GCC using these examples doesn't have much practical significance. Because on the topic of coroutine optimization, GCC and Clang are literally not on the same dimension. What I mainly want to say is that Clang/LLVM's decision in implementing coroutines is extremely correct and visionary.

Of course, there's no free lunch in this world, and this decision also has trade-offs. Specifically:

- Ideally, the LLVM midend should be language-agnostic. However, the decision to implement part of C++20 coroutine semantics in the midend is, in design terms, arguably a kind of false layering. Specifically, it has two directions of drawbacks:
  - To give the midend more language-related information to complete optimization and proper handling, we need to continuously expose interfaces in LLVM to accept information transmitted from the Clang side.
  - We completely lose information about midend optimization results in the frontend (as expected), making it impossible for us to further manipulate or design things related to midend optimization result information at the frontend level. In short, this decision shrinks C++'s design space for coroutines at the language level.

For the former, a specific example is the debuggability of C++20 coroutines. Debug info for other language constructs is generated together when the compiler frontend generates code, and the midend only needs to process these debug info IRs. For coroutines, because the coroutine compilation is not completed when Clang's frontend generates code, the debug info generated by Clang's frontend for coroutines cannot match the actual coroutine code. This affects the debuggability of C++20 coroutines generated by Clang. Although later we generated debug info when performing coroutine transformation in the LLVM midend to alleviate this. Readers interested in the details can refer to (Debugging C++ Coroutines)[https://clang.llvm.org/docs/DebuggingCoroutines.html]. However, with optimization enabled, the debuggability of C++20 coroutines is definitely not as good as GCC's, since GCC doesn't optimize coroutines. Besides debuggability, other issues are more due to lack of information in the midend, so new bugs occasionally appear, though this has limited impact on users in 2026.

For the latter, readers interested in language design may have a stronger feeling. People who want to further design/enhance coroutines often appear constrained when it comes to matters related to coroutine frames. Because the frontend/language basically knows nothing about or delegates all responsibility for the specific composition of coroutine frames to the compiler midend.强行 adding new constraints almost always leads to the compiler generating worse code in some cases. Later in this article, I'll try to provide a specific example.

Note: Another reason LLVM chose to implement coroutines in the midend is that coroutine semantics might be reusable in other languages. Subsequent Swift coroutines and an MLIR dialect were both implemented based on LLVM coroutine capabilities.

# Coroutine Frame Generation & Tips for Reducing Coroutine Frame Size

Clang's algorithm for generating coroutine frames is roughly: for each local variable, temporary variable, and parameter, determine whether its lifetime spans suspend points. If it does, place it on the coroutine frame; otherwise, it won't. In practice, due to optimization, many local variables and parameters in the source code may be optimized away, making the coroutine frame smaller than the sum of local variables, temporary variables, and parameters in the source code.

An important point here is that the compiler midend's current judgment of lifetime mainly comes from the frontend output. And the compiler midend currently only uses lifetime for optimization, not optimizes the lifetime itself.

Using the same example above:

```C++
... same as above

void produce(int *arr) {
    for (int i = 0; i < 500; ++i)
        arr[i] = i * 2;
}

void consume(int *arr) {
    long sum = 0;
    for (int i = 0; i < 500; ++i)
        sum += arr[i];
    // Necessary to make sure these codes won't be
    // optimized out.
    std::cout << "Sum: " << sum << "\n";
}

[[clang::noinline]]
Task my_coro() {
    int arrays[500];
    produce(arrays);
    consume(arrays);
    co_await std::suspend_always{};
    co_return;
}

... same as above
```

In this case, we intuitively think `arrays` should not be placed in the coroutine frame. But the runtime shows:

```C++
Coroutine allocated 2032 bytes
```

The reason is that the lifetime information generated by the frontend can basically be considered as marked near the curly braces (`{}`) surrounding its declaration, i.e.:

```C++
// fake IR
Task my_coro() {
    int arrays[500];
    lifetime_start(&arrays);
    
    produce(arrays);
    consume(arrays);
    co_await std::suspend_always{};
    
    lifetime_end(&arrays);
    co_return;
}
```

So the midend places `arrays` in the coroutine frame, even though from a god's eye view we know this isn't necessary.

## Program Optimization Hacker Tips

When programmers understand this phenomenon, they can try to manually add curly braces (`{}`) to provide the compiler with more accurate lifetime information:

```C++
[[clang::noinline]]
Task my_coro() {
    {
        int arrays[500];
        produce(arrays);
        consume(arrays);
    }
    co_await std::suspend_always{};
    co_return;
}
```

Now the runtime output is:

```C++
Coroutine allocated 24 bytes
```

Due to the complexity of lifetime issues, the midend optimization is unlikely to do sufficient optimization for more complex cases. I believe that even after the compiler midend adds lifetime-shortening optimization, manual optimization like this is still effective for many non-trivial cases.

## Compiler Optimization Opportunities

Conversely, this is another opportunity for the compiler. We can try to add new optimizations/transformations in the midend to shorten lifetime information. We can find each use point of each variable in reverse, then determine whether we can advance the lifetime_end. A key point here might be that we need to be very careful about variable escaping. When a variable escapes, we can no longer make any further assumptions about its lifetime in the midend. At this point, the only thing we can do is follow the frontend's guidance.

## Lifetime-Independent Optimization

When the midend finds that the lifetimes of two variables completely don't overlap, even if both variables need to be placed on the coroutine frame, the compiler can make two variables with non-overlapping lifetimes share the same address:

```C++
[[clang::noinline]]
Task my_coro(bool cond) {
    if (cond) {
        int big_struct[500];
        produce(big_struct);
        co_await std::suspend_always{};
        consume(big_struct);
    } else {
        int big_struct[500];
        produce(big_struct);
        co_await std::suspend_always{};
        consume(big_struct);
    }
    co_return;
}
```

At this point, the runtime output is:
  
```C++
Coroutine allocated 2032 bytes
```

We can see that variables in the two branches share the same memory address.

# HALO & CoroElide

HALO, i.e., [coroutine Heap Allocation eLision Optimization](https://www.open-std.org/jtc1/sc22/wg21/docs/papers/2018/p0981r0.html), refers to the compiler eliminating coroutine dynamic allocation through optimization. The example at the beginning is an application of HALO.

Although the name HALO is widely known, HALO is actually just a compiler optimization feature, not a language feature. This means a programmer cannot write HALO-friendly code by just looking at the Language Spec. The only way a programmer can guarantee HALO is to continuously test different compiler versions.

In LLVM, HALO is called CoroElide. I think the name CoroElide is easier to understand, so I always use this name, even though HALO does look cooler.

## Compiler Automatic CoroElide Conditions & Usage Tips

This section attempts to describe the conditions for automatic compiler CoroElide without diving into compiler implementation details, and to describe the current limitations of compiler CoroElide to users.

In a nutshell, the condition for automatic compiler CoroElide is: the compiler can prove that the lifetime of the coroutine frame converges within the scope of the call site. That is, the coroutine frame is created after entering that scope and is destroyed before exiting that scope.

Let's look at a positive example first:

```C++
#include <coroutine>
#include <iostream>
#include <cstdlib>

struct Task {
    struct promise_type {
        void* operator new(std::size_t size) {
            std::cout << "Coroutine allocated " << size << " bytes\n";
            return ::operator new(size);
        }

        Task get_return_object() { 
            return Task{std::coroutine_handle<promise_type>::from_promise(*this)}; 
        }
        std::suspend_always initial_suspend() noexcept { return {}; }
        std::suspend_always final_suspend() noexcept { return {}; }
        void return_value(int v) { value = v; }
        void unhandled_exception() {}
        
        int value = 0;
    };

    std::coroutine_handle<promise_type> handle;

    ~Task() { 
        if (handle) handle.destroy(); 
    }

    bool done() const {
        return handle && handle.done();
    }
    
    void resume() {
        if (handle)
            handle.resume();
    }
    
    void destroy() {
        if (handle)
            handle.destroy();
    }
    
    int value() {
        if (handle)
            return handle.promise().value;
        
        return 0;
    }
};

Task foo() {
    co_return 43;
}
Task bar() {
    auto f = foo();
    while (!f.done())
        f.resume();
    int value = f.value();
    f.destroy();
    co_return value;
}
int main() {
    auto b = bar();
    while (!b.done())
        b.resume();
    std::cout << b.value() << std::endl;
    b.destroy();
    return 0;
}
```

Compiling and running this case results in:

```
43
```

We can see that dynamic allocations for both coroutine instances, `f = foo()` and `b = bar()`, are successfully optimized away by the compiler. In `bar()`, the compiler can determine that the coroutine frame for `f = foo()` is created at `auto f = foo();` and destroyed at `f.destroy()`. That is, the lifetime of `f = foo()`'s coroutine must converge within `bar()`'s scope. The process for `b = bar()` in the `main` function is similar.

However, this syntax is clearly unacceptable in slightly more complex cases. Let's slightly refactor this case, making `Task` an `awaiter`, and use symmetric transfer:

```C++
#include <coroutine>
#include <iostream>
#include <cstdlib>
#include <future>

struct Task {
    struct promise_type {
        void* operator new(std::size_t size) {
            std::cout << "Coroutine allocated " << size << " bytes\n";
            return ::operator new(size);
        }

        Task get_return_object() { 
            return Task{std::coroutine_handle<promise_type>::from_promise(*this)}; 
        }
        std::suspend_always initial_suspend() noexcept { return {}; }
        struct FinalAwaiter {
            bool await_ready() noexcept { return false; }
            std::coroutine_handle<> await_suspend(std::coroutine_handle<> h) noexcept {
                if (continuation)
                    return continuation;
                return std::noop_coroutine();
            }
            void await_resume() noexcept {}

            std::coroutine_handle<> continuation;
        };
        FinalAwaiter final_suspend() noexcept { return {continuation}; }
        void return_value(int v) { value = v; }
        void unhandled_exception() {}
        
        int value = 0;
        std::coroutine_handle<> continuation;
    };

    std::coroutine_handle<promise_type> handle;

    ~Task() { 
        if (handle) handle.destroy(); 
    }

    bool done() const {
        return handle && handle.done();
    }
    
    bool await_ready() { return false; }
    template <class Promise>
    auto await_suspend(const std::coroutine_handle<Promise> &h) {
        handle.promise().continuation = h;
        return handle;
    }

    int await_resume() {
        int value = handle.promise().value;
        handle.destroy();
        handle = nullptr;
        return value;
    }
};

int syncAwait(Task &&t) {
    std::promise<int> p;
    std::future<int> f = p.get_future();

    auto lambda = [](std::promise<int> p, Task &&t) -> Task {
        p.set_value(co_await t);
        co_return 0;
    };

    auto h = lambda(std::move(p), std::move(t));
    while (!h.done()) h.handle.resume();
    h.handle.destroy();

    return f.get();
}

Task foo() {
    co_return 43;
}
Task bar() {
    co_return co_await foo();
}

int main() {
    std::cout << syncAwait(bar()) << std::endl;
    return 0;
}
```

Then compiling with (Clang >= 18) and running gives:

```
Coroutine allocated 48 bytes
Coroutine allocated 48 bytes
43
```

Two coroutine memory allocations were not optimized by the compiler! (The lambda coroutine dynamic allocation in syncAwait was optimized, as confirmed by examining the assembly)

If we compile with (Clang <= 17) and run, the result becomes:

```
Coroutine allocated 96 bytes
43
```

Memory allocation only occurred once! The memory frame of the `foo()` coroutine instance in `bar()` was optimized to a local variable in `bar()`, and then this local variable was placed in `bar()`'s coroutine frame. This caused `bar()`'s coroutine frame to grow, but the number of memory allocations in the program decreased, which is as expected. The reason `main` cannot optimize `bar()`'s dynamic allocation is that `bar()`'s coroutine instance's end marker is in the lambda function in `syncAwait()`. For the compiler, this is too difficult to handle. Although from a god's eye view we can optimize it, from the compiler implementer's perspective, this is as expected.

So what happened between Clang17 and Clang18? The answer is that since Clang18, to mimic the standard's description that `await_suspend` execution occurs after the coroutine is suspended, `await_suspend` is no longer inlined (more description later).

So for Clang >= 18, the above example's `bar()` code after inline optimization is equivalent to:

```C++
Task bar() {
    Task f;
    f.handle = Task::promise_type::operator new(96); // #1
    /* Initialize f.handle, ignored */
    
    // f.await_ready() is always optimized to false
    f.await_suspend(current_coroutine_handle()); // await_suspend is not inlined
    
    // f.await_resume
    int value = f.handle.promise().value;
    f.handle.destroy(); // #2
    f.handle = nullptr;
    
    // co_return
    current_promise.value = value;
    return;
}
```

Here, #1 creates the `foo()` coroutine frame, and #2 destroys the `foo()` coroutine frame. For humans, it seems obvious that `foo()`'s coroutine frame lifetime converges within `bar()`. But for the compiler, since `f.await_suspend` was not inlined, the compiler must assume that `f.await_suspend` might contain code like:

```C++
auto Task::await_suspend(std::coroutine_handle) {
  ...
  another_handle = this->handle; // Not caring where another_handle comes from
  this->handle = another_handle2; // this->handle changed!
  ...
}
```

When `f.await_suspend` has such code, what #2 destroys is no longer the coroutine frame created in #1. At this point, the coroutine frame created in #1 will no longer be destroyed in `bar()`. Therefore, the compiler cannot prove that `foo()`'s coroutine frame lifetime converges within `bar()`.

So what should we do? The methods are:

1. Use attributes to let programmers automatically mark elidable coroutines. This is described later.
2. Understand compiler alias information analysis to help the compiler prove lifetime convergence information.

Let's describe the second method first:

```C++
#include <coroutine>
#include <iostream>
#include <cstdlib>
#include <future>

struct Task {
    struct promise_type {
        void* operator new(std::size_t size) {
            std::cout << "Coroutine allocated " << size << " bytes\n";
            return ::operator new(size);
        }

        Task get_return_object() {
            return Task{std::coroutine_handle<promise_type>::from_promise(*this)};
        }
        std::suspend_always initial_suspend() noexcept { return {}; }
        struct FinalAwaiter {
            bool await_ready() noexcept { return false; }
            std::coroutine_handle<> await_suspend(std::coroutine_handle<> h) noexcept {
                if (continuation)
                    return continuation;
                return std::noop_coroutine();
            }
            void await_resume() noexcept {}

            std::coroutine_handle<> continuation;
        };
        FinalAwaiter final_suspend() noexcept { return {continuation}; }
        void return_value(int v) { value = v; }
        void unhandled_exception() {}

        int value = 0;
        std::coroutine_handle<> continuation;
    };

    std::coroutine_handle<promise_type> handle;

    ~Task() {
        if (handle) handle.destroy();
    }

    bool done() const {
        return handle && handle.done();
    }

    auto operator co_await() {
        return Awaiter{handle};
    }

    struct Awaiter {
        std::coroutine_handle<promise_type> handle;

        bool await_ready() { return false; }

        template <class Promise>
        auto await_suspend(const std::coroutine_handle<Promise> &h) {
            handle.promise().continuation = h;
            return handle;
        }

        int await_resume() {
            int value = handle.promise().value;
            // handle.destroy();
            handle = nullptr;
            return value;
        }
    };
};

int syncAwait(Task &&t) {
    std::promise<int> p;
    std::future<int> f = p.get_future();

    auto lambda = [](std::promise<int> p, Task &&t) -> Task {
        p.set_value(co_await t);
        co_return 0;
    };

    auto h = lambda(std::move(p), std::move(t));
    while (!h.done()) h.handle.resume();
    h.handle.destroy();

    return f.get();
}

Task foo() {
    co_return 43;
}
Task bar() {
    auto f = foo();
    int v = co_await f;
    f.handle.destroy();
    co_return v;
}

int main() {
    std::cout << syncAwait(bar()) << std::endl;
    return 0;
}
```

After compiling and running with Clang >= 18, we see:

```
Coroutine allocated 96 bytes
43
```

The main difference from the previous version is in the `Task` awaiter implementation:

```C++
    auto operator co_await() {
        return Awaiter{handle};
    }

    struct Awaiter {
        std::coroutine_handle<promise_type> handle;

        bool await_ready() { return false; }

        template <class Promise>
        auto await_suspend(const std::coroutine_handle<Promise> &h) {
            handle.promise().continuation = h;
            return handle;
        }

        int await_resume() {
            return handle.promise().value;
            // handle.destroy();
            // handle = nullptr;
            return value;
        }
    };
```

At this point, `bar()`'s implementation is equivalent to:

```C++
Task bar() {
    Task f;
    f.handle = Task::promise_type::operator new(96); // #1
    /* Initialize f.handle, ignored */
    
    Task::Awaiter awaiter{f.handle}; // Copy construction
    // awaiter.await_ready() is always optimized to false
    awaiter.await_suspend(current_coroutine_handle()); // await_suspend is not inlined
    
    // awaiter.await_resume
    int value = awaiter.handle.promise().value;
    
    // f.~Task()
    // Compiler can analyze that f.handle must not be null.
    f.handle.destroy(); #2
    
    // co_return
    current_promise.value = value;
    return;
}
```

At this point, because `awaiter` only has a copy of `f.handle`, and the addresses of `f` and `f.handle` have not been leaked through any means, the compiler can determine that regardless of `awaiter.await_suspend`'s implementation, the `f.handle` at #2 must point to the same coroutine instance as `f.handle` at #1. Therefore, the compiler can complete the optimization of `foo()`'s coroutine dynamic allocation in `bar()` as expected, optimizing it to a local variable in `bar()`!

### Compiler Optimization Opportunities

Here, the compiler can give users a function attribute or member variable attribute to let the programmer hint to the compiler that `await_suspend` will not modify `this->handle` or `this->handle`'s value will not change (except setting to nullptr). This further enables Coro Elide.

## [[clang::coro_await_elidable]]

To facilitate library authors in applying coro elide optimization, clang introduced the `[[clang::coro_await_elidable]]` class attribute. Its semantics are: as long as the return value of a coroutine function returning this class type is **immediately** co_awaited, the programmer guarantees to the compiler that the coroutine instance corresponding to this coroutine function can be elided.

For example:

```C++
class [[clang::coro_await_elidable]] Task { ... };

Task foo();
Task bar() {
  co_await foo(); // foo()'s coroutine frame on this line is elidable
  auto t = foo(); // foo()'s coroutine frame on this line is NOT elidable
  co_await t;
}
```

In `bar()`, the first line's `co_await foo()` immediately co_awaits `foo()`'s return value, so the compiler can consider this coroutine instance elidable. The second line's `t = foo()` is not immediately co_awaited, so it doesn't meet the previous convention.

The "immediately" co_awaited terminology might be strange to users. This is mainly due to implementation difficulties. Doing data flow analysis on the frontend AST is difficult, so as a compromise, we chose the "immediately" co_awaited description, which is easy to implement and relatively easy for users to understand.

### Compiler Optimization Opportunities

Although frontend data flow analysis is difficult, it's not impossible. If we can determine in the frontend whether a coroutine instance **always** will be co_awaited, we can extend the semantics of this attribute. At the same time, we can also make clang-tidy security checks more flexible. Currently, [cppcoreguidelines-avoid-capturing-lambda-coroutines](https://clang.llvm.org/extra/clang-tidy/checks/cppcoreguidelines/avoid-capturing-lambda-coroutines.html) disallows any captures in coroutine lambdas, which is overly strict. Many cases would be fine if we could ensure they are always co_awaited.

## [[clang::coro_await_elidable_argument]]

The `[[clang::coro_await_elidable_argument]]` attribute is mainly an extension for whenall semantics, aiming to allow coroutines like `foo()`, `bar()`, `zoo()` in `co_await whenall(foo(), bar(), zoo())` to also be optimized by Coro Elide.

Usage:

```C++
template <typename T>
class [[clang::coro_await_elidable]] Task { ... };

template <typename... T>
class [[clang::coro_await_elidable]] WhenAll { ... };

// `when_all` is a utility function that composes coroutines. It does not
// need to be a coroutine to propagate.
template <typename... T>
WhenAll<T...> when_all([[clang::coro_await_elidable_argument]] Task<T> tasks...);

Task<int> foo();
Task<int> bar();
Task<void> example1() {
  // `when_all`, `foo`, and `bar` are all elide safe because `when_all` is
  // under a safe elide context and, thanks to the [[clang::coro_await_elidable_argument]]
  // attribute, such context is propagated to foo and bar.
  co_await when_all(foo(), bar());
}

Task<void> example2() {
  // `when_all` and `bar` are elide safe. `foo` is not elide safe.
  auto f = foo();
  co_await when_all(f, bar());
}

Task<void> example3() {
  // None of the calls are elide safe.
  auto t = when_all(foo(), bar());
  co_await t;
}
```

## Is coro eliding always an optimization?

### Background Story

We enhanced CoroElide's capabilities extensively internally early on and expected to gain positive performance benefits by reducing memory allocation counts. However, every time we ran performance tests, we actually observed performance regressions rather than improvements in most cases.

### Analysis

#### Memory Usage

In practice, coroutines and async high-concurrency programming are often linked. So we can expect that for an async chain entry, we always need dynamic allocation:

```C++
Task zoo() {
    something
}
Task foo() {
    ...
    co_await zoo();
    ...
}
Task bar() {
    ...
    co_await foo();
    ...
}
void entry() {
    bar().start(callback); // This is a non-blocking call! We must dynamically allocate
    ...
}
```

If the coroutine frame dynamic allocations for both foo() and zoo() are elided, the memory allocated in `entry` is:

```
┌─────────────────────────────────────────┐
│           bar() coroutine frame          │
│  ┌───────────────────────────────────┐  │
│  │        foo() coroutine frame       │  │
│  │  ┌─────────────────────────────┐  │  │
│  │  │       zoo() coroutine frame  │  │
│  │  │      parameters/local vars + coroutine state  │  │
│  │  └─────────────────────────────┘  │  │
│  │                                   │  │
│  │   parameters/local vars + coroutine state       │  │
│  └───────────────────────────────────┘  │
│                                         │
│   parameters/local vars + coroutine state                 │
└─────────────────────────────────────────┘
```

We need to allocate `sizeof(bar's self states) + sizeof(foo's self states) + sizeof(zoo's self states)` of memory at once.

If both foo() and zoo()'s coroutine frames are dynamically allocated on demand, then before or after foo() and zoo() execute, the actual memory usage actually decreases:

```
Memory
  ▲  ┌────────────────────────────────────────────────────────────┐
  │  │█████████████████████████████ bar + foo + zoo ██████████████│
  │  │████████████████████████████████████████████████████████████│
  │  │████████████████████████████████████████████████████████████│
  │  │████████████████████████████████████████████████████████████│
  └──┴────────────────────────────────────────────────────────────┴──▶ time
     bar.start                                                      bar.end

```

```
Memory
  ▲  ┌────────────────────────────────────────────────────────────┐
  │  │                    ███████████████████                     │
  │  │                    ███████████████████                     │
  │  │         ███████████████████████████████████████            │
  │  │████████████████████████████████████████████████████████████│
  └──┴────────────────────────────────────────────────────────────┴──▶ time
     bar.start foo.start() zoo.start     zoo.end   foo.end         bar.end
```

As the coroutine chain grows, it may well cause the memory peak to be higher than before, and keep memory usage elevated for longer.

##### Compiler Optimization Opportunities

We may need to add new dimensions to Profiling information so the compiler can determine the length of coroutine instance lifetimes. For coroutine instances with sufficiently long lifetimes, we can consider that eliding their dynamic allocation is helpful. For coroutine instances with very short lifetimes, we can lean toward not eliding. The size of the coroutine frame should also be one of the factors.

#### Hot/Cold Information

Another issue with LLVM's previous Coro Elide was that LLVM would perform CoroElide "optimization" whenever it found a coroutine met the conditions at compile time, but this is not always actually an optimization.

For example:

```C++
Task foo() { ... }
Task bar(bool cond) {
    if (cond) {
        ...
        co_await foo();
        ...
    }
    
    ...
}
```

If in practice `cond` is always or almost always false, then the compiler performing CoroElide optimization on `co_await foo();` in the if statement is a negative optimization. This only causes `bar()`'s coroutine frame to grow without any positive effect.

In Clang22, we added hot/cold information-based judgment to CoroElide. Without Profiling information, static analysis is used to determine hot/cold. The principle is roughly: if the compiler is in a loop, it's considered hot; if it's in an if statement and not in a loop, it's considered cold. Then CoroElide won't be performed.

##### Compiler Optimization Opportunities

However, hot/cold information judgment is currently only enabled when using `[[clang::coro_await_elidable_argument]]`. For automatic CoroElide that doesn't rely on programmer annotations, this feature hasn't been used yet.

## General Limitations of CoroElide & Standardization

Besides the limitations mentioned above, an important limitation of CoroElide is that the compiler must see the function body of the coroutine being elided at compile time. Only then can the compiler calculate the coroutine frame size and convert the coroutine frame to a local variable.

For current compilers, this is generally achieved through inlining. Later, it might be possible to achieve this through cross-function analysis. However, in any case, under the current single-process single-file compilation model, the compiler cannot obtain the specific information of coroutine implementations in other unreachable compilation units, making elide impossible.

Although theoretically similar goals could be achieved through cross-TU analysis or LTO-like techniques, these limitations make it very difficult for us to promote CoroElide-related features in the C++ Standards Committee.

## Language-Coordinated Coroutine Memory Allocation Optimization Scheme

### Specifying Coroutine Frame Upper Size Limit, Frontend-Opaque, Explicit Coroutine Frame Type

The author of this idea is Lewis Baker. He has too many papers, so I was lazy and didn't go through them in detail. Here I'll describe based on memory the approach that impressed me most.

The general idea is to have the compiler frontend calculate an upper size limit for a coroutine frame. Then we can allocate memory in advance based on this upper size limit. Then provide a way for programmers to use the pre-allocated memory as the coroutine frame's memory space without modifying the function signature.

For example:

```C++
Task foo(...) { ... }
Task bar(...) { ... }

...

auto frame = decl_coro_frame_type(bar(a, b, c));
auto t = call_coro_with_specified_frame(bar(a, b, c), frame);

auto size = get_coro_frame_size_uplimit(foo(a, b, c));
void *buffer = personlized_allocator(size);
auto f = call_coro_with_specified_frame(foo(a, b, c), buffer);
```

These two examples illustrate using compiler frontend primitives to get the coroutine frame size upper limit, then using local variables or calling custom allocators to allocate coroutine frame memory.

This idea is implementable, but:

1. It still requires the coroutine implementation to be visible in the current TU.
2. It requires the compiler to disable current CoroElide optimization. Otherwise, there's no point in discussing coroutine frame size upper limits. See the description of memory usage in the previous section. CoroElide may cause coroutine frame bloat.
3. For compilers like Clang that rely on midend optimization, such a frontend way of getting frame size upper limit may well result in the obtained size being much larger than actually needed (see the Clang vs GCC section earlier). For users, this could very well be a negative optimization.

Points 2 and 3 are what I described earlier: Clang/LLVM's choice in implementing coroutines conversely limits the design space for coroutine language.

### Coroutine Elided Allocation Hook && Stacked Allocation

Idea from MicroCai, https://github.com/alibaba/async_simple/discussions/398

The general idea is to provide a new set of allocation and deallocation interfaces in the promise type. When the compiler decides to elide, instead of simply converting the coroutine frame to a local variable, it tries to use the allocation and deallocation interfaces provided in the promise type. The clever part here is that these promise type allocation and deallocation allocators can use efficient stack-based memory allocators. When we need allocation, we just increase the stack pointer. Of course, if stack space is insufficient, we need to request more.

This design has the following disadvantages:

1. It assumes elided coroutine instances will be constructed and destructed sequentially. This assumption is stronger than what current CoroElide's implementation requires. For example, for the Whenall scenario mentioned earlier, directly applying this design would be dangerous.
2. This design only targets classic Task-type coroutines. C++20 coroutines allow users very strong extensibility in design. But this design only targets one specific type of coroutine. Of course, this type of coroutine is indeed the mainstream.

After understanding the above limitations of static CoroElide, this method's advantages are also obvious:

1. Avoids hot/cold path issues.
2. Uses local efficient stacked allocation, avoiding relatively inefficient global memory allocation.
3. Can avoid issues of excessive memory overhead caused by premature allocation.

I have also implemented a draft version in the compiler: https://github.com/ChuanqiXu9/llvm-project/tree/StackedAllocation 

The related library implementation is at: https://github.com/alibaba/async_simple/commit/a13f69eb408fb7e7ce25ca1f75f42e84931ab71c. Interested friends can directly try this branch and test it in their own coroutine library based on its implementation.

I plan to find time to test the effect on large applications. Interested friends can also try it themselves and run some small benchmarks.

## Summary

CoroElide is the most famous optimization in coroutine optimization. Due to the complexity of dynamic allocation, it's difficult to statically determine the optimal memory allocation method. (Friends who have done memory allocation tuning may understand this better). In the long run, this field has great potential in compiler, library, and compiler-library collaborative implementation directions.

1. Automatic CoroElide has changed due to await_suspend changes, resulting in many originally auto-elidable codes not being automatically elided after Clang18. After Clang18, some extra hacks are needed to write auto-elide-friendly code. Compiler developers also need to provide additional attributes for users to inform the compiler that `await_suspend` will not modify `this->handle`.
2. Newer Clang versions provide `[[clang::coro_await_elidable]]` attribute, providing deterministic coroutine elide opportunities. However, due to frontend architecture reasons, elision can only be implemented for immediately awaited coroutine objects. Later, the compiler frontend should either strengthen data flow analysis in this area, or still need automatic Elide functionality based on midend capabilities.
3. Static CoroElide is not always an optimization. There may be many compiler opportunities here for Profiling-based optimization, or the possibility of optimization hints that let users provide high-level information more flexibly.
4. Compiler and library collaborative optimization has great potential.

# noexcept

## Background Story

After coroutines were announced into the standard, we started trying to apply coroutines. At that time, due to relatively primitive compiler and library support, and because async programming lifetime issues were difficult to investigate, we didn't directly switch the original sync chain to async chain for risk considerations. Instead, we considered using macros to make one set of code flexibly switchable between coroutine-based sync chain and coroutine-based async chain.

When comparing coroutine-based sync chain with the original sync chain, we found a significant performance regression. We quickly identified it was due to coroutine allocation. Then, based on compiler implementation, we optimized the coroutine library implementation to allow auto-elide (at that time Clang version was still low). However, we still found significant performance differences. Eventually, we identified that it was caused by coroutines being inserted into try-catch.

## Automatic noexcept Propagation

For

```C++
void bar();
[[clang::noinline]] // For better readability of generated code
void foo() {
    try {
        bar();
    } catch(...) {}
}
void zoo() {
    try {
        foo();
    } catch (...) {
        
    }
}
```

We should hope the compiler can automatically (in the midend) mark `foo()` as noexcept, so that the useless try catch can be eliminated in `zoo()`.

However, Clang currently cannot do this:

```
clang++ -O3 -S -emit-llvm -o - %s
```

```

  invoke void @foo()()
          to label %5 unwind label %1, !dbg !25

  %2 = landingpad { ptr, i32 }
          catch ptr null, !dbg !27
  %3 = extractvalue { ptr, i32 } %2, 0, !dbg !27
  %4 = tail call ptr @__cxa_begin_catch(ptr %3) #3, !dbg !28
  tail call void @__cxa_end_catch(), !dbg !29
  br label %5, !dbg !29

5:
  ret void, !dbg !31

```

After discussion (https://reviews.llvm.org/D108905), this seems to be a "hole" in the language standard (not confirmed). The conclusion is that it depends on what behavior the program should be in when a destructor of an exception throws an exception.

For example, if `bar()`'s implementation is:

```C++
struct E {
  ~E() noexcept(false) {
    throw 3;
  }
};
void bar() {
  throw E();
}
```

Then even if `foo()`'s implementation has `catch(...)` to try to catch all exceptions, the compiler still cannot determine that `foo()` will definitely not throw exceptions.

As far as I know, the Standards Committee doesn't seem to have reached a conclusion on whether destructors are allowed to throw exceptions, or what behavior is when a destructor's destructor throws an exception. But from the Clang community's perspective, we hope not to casually change existing behavior. So we didn't do further optimization. But since destructors throwing exceptions from exceptions are indeed relatively rare cases, we provided `-fassume-nothrow-exception-dtor` to let users inform the compiler that there are currently no exceptions where destructors throw exceptions, allowing the compiler to optimize confidently:

```
clang++ -O3 -S -emit-llvm -o - %s -fassume-nothrow-exception-dtor
```

```
define dso_local void @zoo()() local_unnamed_addr #2 !dbg !24 {
  tail call void @foo()(), !dbg !25
  ret void, !dbg !27
}
```

We can see very obvious optimization effects.

## Relationship with Coroutines

The above discussion is interesting, but what's the relationship with coroutines?

The answer is that the language design guarantees every coroutine is wrapped in try-catch:

```C++
{
  promise-type promise promise-constructor-arguments ;
  try {
    co_await promise.initial_suspend() ;
    function-body
  } catch ( ... ) {
    if (!initial-await-resume-called)
        throw ;
    promise.unhandled_exception() ;
  }
  final-suspend :
    co_await promise.final_suspend() ;
}
```

Where `final_suspend()` is guaranteed by the language not to throw. So if the coroutine library author can guarantee that `promise.unhandled_exception()` will not throw, and `co_await promise.initial_suspend()` and before (coroutine frame allocation and promise construction) will not throw, then all coroutines of this coroutine type can be considered not throwing. At this point, with `-fassume-nothrow-exception-dtor`, we can automatically add `noexcept` to every coroutine. This triggers more optimization.

# Coro RVO

Coroutines don't have traditional return statements, but calling a coroutine function still returns an object to the caller through `promise_type.get_return_object()`.

In Clang, we implemented RVO optimization for this return, i.e., directly constructing the coroutine in the caller's return slot to avoid repeated construction and passing overhead.

However, due to implementation limitations, currently RVO in coroutines is only enabled when the coroutine return type matches `promise_type.get_return_object()`'s type.

# await_suspend Execution Occurs After Coroutine Suspension

The standard states:

> [expr.await]p5.1
>   If the result of await-ready is false, the coroutine is considered suspended. 

This means when `await_suspend()` function executes, its coroutine is already suspended, i.e., it has stopped running.

In the currently compiler-generated code, `await_suspend()` actually still executes in its coroutine. We execute `await_suspend()`, and after it completes, based on its return value, determine whether to continue or suspend (return). The main purpose of this is better performance: besides being able to inline `await_suspend()`, when `await_suspend()` determines execution still needs to continue, we don't need to actually pop and push the stack; we just execute directly.

But in practice, we found multiple related bugs. The most important is the usage of destroying a coroutine in `await_suspend()`. According to the standard, this usage is没有问题 because the coroutine is already considered suspended when `await_suspend()` executes. But with LLVM's previous implementation method, `await_suspend()` actually still executes while the coroutine is considered still running.

To fix this bug, we no longer inline `await_suspend()` to mimic the standard's wording. This is also why automatic elide is no longer possible starting from Clang18, as mentioned earlier.

But note, we are still just mimicking the standard's wording; we haven't actually moved `await_suspend()` out of the coroutine.

# initial_suspend Not Being always_suspend Generates More Code and Blocks CoroElide Optimization

Currently, LLVM's compilation process for coroutines with `initial_suspend` not being `always_suspend` is relatively unfriendly, generating a lot of additional code.

The reason is that LLVM's current design for compiling coroutines splits a coroutine into three parts: Ramp, Resume, and Destroy:

- Ramp. Responsible for coroutine memory allocation, Promise construction, and coroutine frame initialization. Corresponds to the original coroutine function.
- Resume. Jumps to different suspend points based on the current state of the coroutine. Corresponds to `std::coroutine_handle<>::resume()`.
- Destroy. Destroys the coroutine frame. Corresponds to `std::coroutine_handle<>::destroy()`.

Currently, the division between Ramp and Resume is based on the first place that can be determined at compile time to be a point of inevitable suspension, i.e., the `initial_suspend` that is `always_suspend`. The current Ramp function is very small when `initial_suspend` is `always_suspend`, which is very beneficial for inlining.

```C++
  tail call fastcc void %35(ptr nonnull %34)
  ret void

36:
  %37 = landingpad { ptr, i32 }
          cleanup
  tail call void @operator delete(void*, unsigned long)(ptr noundef nonnull %5, i64 noundef 56) #23
  resume { ptr, i32 } %37
}
```

This actually contains all the content of the `Resume` function! In most cases, the main content of a coroutine function is in the `Resume` function. Duplicating the `Resume` in the Ramp function means coroutine-related code volume swells! And in large projects, code volume swelling often also leads to performance degradation.

Besides volume, another downside is that the increased coroutine Ramp function hinders its chance of being inlined. This further hinders opportunities for CoroElide.

## Compiler Optimization Opportunities

For coroutines where `initial_suspend` is not `suspend_always`, we should consider forcefully inserting a call to the Resume function at `initial_suspend`. This will add one more function call, but can greatly reduce code volume and enable CoroElide.

# [[clang::coro_only_destroy_when_complete]]

Since a coroutine may be destroyed after any suspend point, the coroutine's destroy function needs to prepare for different suspend points.

For example:

```C++
A foo() {
  dtor d;
  co_await something();
  dtor d1;
  co_await something();
  dtor d2;
  co_return 43;
}
```

The compiler-generated destroy function might be (pseudocode):

```C++
void foo.destroy(foo.Frame *frame) {
  switch(frame->suspend_index()) {
    case 1:
      frame->d.~dtor();
      break;
    case 2:
      frame->d.~dtor();
      frame->d1.~dtor();
      break;
    case 3:
      frame->d.~dtor();
      frame->d1.~dtor();
      frame->d2.~dtor();
      break;
    default: // coroutine completed or haven't started
      break;
  }

  frame->promise.~promise_type();
  delete frame;
}
```

But as a designer of coroutine library A, if I know that coroutine A's design does not allow Cancel semantics, i.e., any coroutine will only be destroyed after completion, the above generated code is very inefficient. At this point, we can add the class attribute `[[clang::coro_only_destroy_when_complete]]` to A to convey this information. At this point, the compiler can generate:

```C++
void foo.destroy(foo.Frame *frame) {
  frame->promise.~promise_type();
  delete frame;
}
```

## Compiler Optimization Opportunities

1. The information in `[[clang::coro_only_destroy_when_complete]]` might be too strong. Because many coroutine libraries might support Cancel semantics now. We might need to add call-site or suspend-point level attributes, allowing users to inform the compiler that this suspend will definitely return, enabling more fine-grained optimization.

For example:

```C++
CancellableTask foo() {
    
    [[clang::must_resume]] co_await something();
    
    ...
}
```

We cannot mark `CancellableTask` with `[[clang::coro_only_destroy_when_complete]]`. But we can inform the compiler that `[[clang::must_resume]] co_await something();` is safe to optimize here.

2. Besides optimizing the destroy function, `[[clang::coro_only_destroy_when_complete]]` itself carries the information that every suspend point of this type's coroutines will eventually be resumed. This information might be used for optimization, but currently it hasn't been well utilized.

# CIR

CIR attempts to solve:

1. The problem of lacking data flow analysis in the frontend.
2. The problem of lacking language information in the midend.

by establishing a new abstraction layer between Clang AST and LLVM IR.

At first glance, CIR seems very suitable for coroutine generation, avoiding the false layering problem we mentioned earlier. But in reality, CIR for coroutines should still only do analysis. The real code generation must be delayed to where coroutine frames are currently generated. The reason is that the current position in LLVM where coroutine frames are generated is relatively late, to allow coroutine code to fully enjoy the benefits of midend optimization. If we move the coroutine code generation function forward, it would almost certainly cause performance regression.

But CIR should still be helpful for coroutine performance. Through CIR-level analysis, we can insert more optimization hints to guide LLVM in optimizing coroutine code.

# Summary

For users:

1. Using Clang can provide significant performance improvement.
2. Using `{}` to refine variable lifetimes can help reduce coroutine frame size.
3. After confirming that no exceptions where destructors throw exceptions exist in the codebase, you can enable `-fassume-nothrow-exception-dtor` optimization.
4. Coroutine return types should try to match `get_return_object`.

For coroutine library authors:

1. You can try to write CoroElide-friendly code (avoid `this` and `this->handle` escaping).
2. CoroElide is not always an optimization. But there is still a lot of optimization space and opportunity.
3. Stacked allocation is worth trying.
4. If coroutines won't be cancelled, consider using `[[clang::coro_only_destroy_when_complete]]`.
5. `initial_suspend` should preferably be `std::suspend_always`.

For compiler developers:

1. Consider reducing coroutine local variable lifetime information. Escaping analysis needs attention.
2. Profiling-based optimization for coroutines may have great opportunity. For example, CoroElide optimization combined with Profiling information.
3. There are many optimization opportunities in coordinating with the language frontend, allowing users to provide more information for further optimization.
